# ANTLR4 → antlr-ng Error Handling Migration Investigation

## Summary

This document investigates the differences in error handling between the current
`antlr4` (v4.13.2) runtime and the `antlr4ng` (v3.x) / `antlr-ng` runtime,
with specific focus on the `.exception` property on `ParserRuleContext` which has
been **removed** in antlr4ng.

## Key Finding: `.exception` Property Removed

### In antlr4 (current)

`ParserRuleContext` has an `exception?: RecognitionException` property that gets
set in **two places**:

1. **Generated parser code (try/catch blocks):** The ANTLR4 code generator
   (`TypeScript.stg` template) wraps every rule method in a try/catch that
   assigns the exception to the local context:

   ```typescript
   // Generated by ANTLR4 for each parser rule:
   try {
       // ... rule body ...
   } catch (re) {
       if (re instanceof RecognitionException) {
           localctx.exception = re;           // <-- THIS LINE
           this._errHandler.reportError(this, re);
           this._errHandler.recover(this, re);
       } else {
           throw re;
       }
   }
   ```

2. **`BailErrorStrategy.recover()`:** Walks up the entire context chain setting
   `.exception` on every ancestor before throwing:

   ```typescript
   // In BailErrorStrategy:
   recover(recognizer, e) {
       let ctx = recognizer._ctx;
       while (ctx !== null) {
           ctx.exception = e;  // <-- Set on every parent context
           ctx = ctx.parentCtx;
       }
       throw new ParseCancellationException(e);
   }
   ```

### In antlr4ng / antlr-ng (new)

- `ParserRuleContext` class has **NO `.exception` property** at all
- The generated parser code (from `antlr-ng` TypeScript.stg template) does
  **NOT** assign `localctx.exception = re` in catch blocks:

  ```typescript
  // Generated by antlr-ng for each parser rule:
  try {
      // ... rule body ...
  } catch (re) {
      if (re instanceof antlr.RecognitionException) {
          this.errorHandler.reportError(this, re);   // No localctx.exception assignment
          this.errorHandler.recover(this, re);
      } else {
          throw re;
      }
  }
  ```

- `BailErrorStrategy` also does **NOT** walk the context chain to set
  `.exception`. It simply wraps and rethrows as `ParseCancellationException`.
- The JSDoc comment on `BailErrorStrategy` still references
  `ParserRuleContext.exception` but this is stale documentation copied from the
  Java implementation.

**From the antlr4ng 3.0.0 release notes:**

> "Neither `BailErrorStrategy` nor `ParserRuleContext` track the original
> exception in error cases anymore. Instead it is wrapped in the exception thrown
> (for bailing out) or passed to the error listeners."

This was an intentional design decision, not an oversight.

## Impact on This Codebase

### Affected Code Locations

#### 1. `src/formatting/formattingHelpers.ts` (lines 134-146)

```typescript
if (tree.exception) {
    const idx = tree.exception.offendingToken.tokenIndex;
    const errorTokens = tokens.tokens.slice(idx);
    // ... uses exception to find unparseable segments
}
```

**Purpose:** Detects whether the root parse tree encountered an error, and if so,
extracts the unparseable portion of the query starting from the offending token.

**Migration impact:** HIGH - This is the primary mechanism for the formatter to
identify unparseable query segments. Without `.exception`, the formatter cannot
determine where parsing failed.

#### 2. `src/autocompletion/completionCoreCompletions.ts` (lines 776, 794)

```typescript
.findLast((x) =>
    (x instanceof RelationshipPatternContext ||
     x instanceof NodePatternContext) &&
    x.exception === null  // <-- Filters out error contexts
)
```

**Purpose:** Filters out parse tree contexts that have errors when searching for
valid completions. Only considers contexts where parsing succeeded.

**Migration impact:** MEDIUM - Used as a quality filter in autocompletion. Without
it, completions might be suggested based on incorrectly-parsed contexts.

#### 3. `src/autocompletion/schemaBasedCompletions.ts` (lines 356, 446)

```typescript
if (child.exception === null) {
    return true;
}
```

**Purpose:** Same pattern as above - filters `RelationshipPatternContext` and
`NodePatternContext` children to only process those that parsed without errors.

**Migration impact:** MEDIUM - Same concern as completionCoreCompletions.

### Other API differences noted

| Feature | antlr4 | antlr4ng |
|---------|--------|----------|
| `ParserRuleContext.exception` | `RecognitionException \| undefined` | **Removed** |
| `ParserRuleContext.parentCtx` | `ParserRuleContext \| undefined` | Renamed to `.parent` (`ParserRuleContext \| null`) |
| `Parser._ctx` | Available (private) | Renamed to `.context` (public) |
| `Parser._errHandler` | Available | Renamed to `.errorHandler` |
| `Parser._parseListeners` | Array assignment | Use `addParseListener()` / `removeParseListeners()` |
| Error listener interface | `ErrorListener<T>` | `ANTLRErrorListener` (no generic param on class) |
| `ErrorListener.syntaxError` | Has generic `<T>` | Simplified signature |

## Migration Strategies

### Strategy 1: Custom Error Strategy (Recommended)

Create a custom `DefaultErrorStrategy` subclass that re-introduces the
`.exception` property assignment. This is the most backward-compatible approach:

```typescript
import { DefaultErrorStrategy, Parser, RecognitionException, ParserRuleContext } from "antlr4ng";

// Extend ParserRuleContext to add the exception property back
declare module "antlr4ng" {
    interface ParserRuleContext {
        exception?: RecognitionException | null;
    }
}

class ExceptionTrackingErrorStrategy extends DefaultErrorStrategy {
    reportError(recognizer: Parser, e: RecognitionException): void {
        // Set exception on the current context (mimics old ANTLR4 behavior)
        if (recognizer.context) {
            (recognizer.context as any).exception = e;
        }
        super.reportError(recognizer, e);
    }

    recover(recognizer: Parser, e: RecognitionException): void {
        // Set exception on the current context
        if (recognizer.context) {
            (recognizer.context as any).exception = e;
        }
        super.recover(recognizer, e);
    }
}
```

**Pros:** Minimal code changes needed, preserves existing logic
**Cons:** Relies on monkey-patching / module augmentation, may miss cases where
generated code previously set `.exception` before calling reportError

### Strategy 2: Custom Generated Code via Grammar Actions

Add `@parser::members` or per-rule `catch` clauses in the grammar to manually
set `.exception`:

```antlr
@parser::members {
    // Nothing needed here, but catch clauses below will set exception
}

// In each rule that needs it, or use a global catch:
rule
    : /* alternatives */
    ;
    catch [re] {
        this.context.exception = re;
        this.errorHandler.reportError(this, re);
        this.errorHandler.recover(this, re);
    }
```

**Pros:** Works within the code generation framework
**Cons:** Requires modifying every grammar rule that needs it, very verbose

### Strategy 3: Post-Parse Tree Walk (for filtering use cases)

For the autocompletion use cases that only check `x.exception === null`, replace
the check with an ErrorNode-based detection:

```typescript
import { ErrorNode } from "antlr4ng";

function hasDirectErrorNode(ctx: ParserRuleContext): boolean {
    return ctx.children.some(child => child instanceof ErrorNode);
}

// Instead of: x.exception === null
// Use:        !hasDirectErrorNode(x)
```

**Pros:** Uses the built-in parse tree error representation, no monkey-patching
**Cons:** Semantically slightly different - ErrorNode presence vs exception
assignment. May not perfectly match old behavior for all error recovery scenarios.

### Strategy 4: Wrap Parser Rule Calls (for formatting use case)

For the formatting use case that checks `tree.exception` on the root context,
wrap the parse call:

```typescript
function parseWithExceptionTracking(parser: CypherCmdParser): StatementsOrCommandsContext {
    try {
        return parser.statementsOrCommands();
    } catch (e) {
        if (e instanceof RecognitionException) {
            // The root context is available
            const ctx = parser.context as StatementsOrCommandsContext;
            (ctx as any).exception = e;
            return ctx;
        }
        throw e;
    }
}
```

**Pros:** Targeted fix for the specific use case
**Cons:** Only catches exceptions that propagate to the caller, doesn't handle
cases where error recovery succeeds within the parser

### Strategy 5: Error Listener with Context Map

Use the error listener's `syntaxError` callback to build a
`Map<ParserRuleContext, RecognitionException>` during parsing, then consult this
map when walking the tree:

```typescript
import { ParserRuleContext, RecognitionException, Token } from "antlr4ng";

class ExceptionTrackingListener implements ANTLRErrorListener {
    readonly exceptions = new Map<ParserRuleContext, RecognitionException>();

    syntaxError(recognizer: Recognizer, offendingSymbol: Token | null,
                line: number, charPositionInLine: number,
                msg: string, e: RecognitionException | null): void {
        if (e && e.ctx) {
            this.exceptions.set(e.ctx, e);
        }
    }
}

// Usage:
const listener = new ExceptionTrackingListener();
parser.addErrorListener(listener);
const tree = parser.statementsOrCommands();

// Instead of: tree.exception
// Use:        listener.exceptions.get(tree)

// Instead of: x.exception === null
// Use:        !listener.exceptions.has(x)
```

**Pros:** Clean separation, no monkey-patching, fully typed, directly uses the
error information that antlr4ng already provides to listeners
**Cons:** Requires threading the listener/map through to all call sites that need
it. The `RecognitionException.ctx` property gives the innermost context where the
error occurred, not necessarily all ancestor contexts (unlike old
BailErrorStrategy behavior).

## Recommended Approach

Use a **combination of Strategy 1 and Strategy 3**:

1. **For the formatting use case** (`formattingHelpers.ts`): Implement
   `ExceptionTrackingErrorStrategy` that sets `.exception` on the context during
   error reporting. This preserves the exact semantics needed to identify
   unparseable segments.

2. **For the autocompletion use cases** (`completionCoreCompletions.ts`,
   `schemaBasedCompletions.ts`): Migrate to ErrorNode-based detection using
   Strategy 3. This is more idiomatic for antlr4ng and doesn't rely on the
   removed `.exception` property. The codebase already has a `hasErrorNodesUnder()`
   utility in `parserWrapper.ts` that demonstrates this pattern.

3. **Long-term:** Consider whether the error recovery strategy itself needs
   rethinking. The `DefaultErrorStrategy` in antlr4ng still performs the same
   single-token insertion/deletion and resync recovery. The error information is
   still available through error listeners - it's just no longer stored on
   context nodes.

## Other Error Detection Approaches in antlr4ng

Beyond the strategies above, there are simpler checks that may suffice for some
use cases:

- **`parser.numberOfSyntaxErrors`** — A simple count of all errors after
  parsing. Useful for quick "did parsing succeed?" checks.
- **`context.stop` token** — If a rule was interrupted by an error, the `stop`
  token may be `null` or point to an unexpected position.
- **`ErrorNode` in children** — During error recovery, the parser adds
  `ErrorNode` children (via `addErrorNode()`) instead of `TerminalNode`. You can
  detect these by checking `child instanceof ErrorNode` or by implementing
  `visitErrorNode()` in a listener/visitor.
